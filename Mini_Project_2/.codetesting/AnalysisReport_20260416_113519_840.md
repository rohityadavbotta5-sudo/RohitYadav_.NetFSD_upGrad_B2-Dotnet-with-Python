# Test Failures due possible code bugs

## EMS.API.UnitTests.csproj - GetAll_WhitespaceSearch_NoMatches_ReturnsZeroData
- **Confidence**: High
- **Test File**: ..\EMS.API.UnitTests\Controllers\EmployeeControllerTests.cs
- **Bug Location**: ..\EMS.API.UnitTests\Controllers\EmployeeControllerTests.cs@820

### Analysis
The test throws System.MissingMethodException when calling UseInMemoryDatabase. The stack trace shows the failure occurs inside Microsoft.EntityFrameworkCore.InMemory's UseInMemoryDatabase extension which calls into EF Core diagnostics helper AbstractionsStrings.ArgumentIsEmpty. A MissingMethodException of this kind indicates an assembly/version mismatch between EF Core packages at runtime (for example Microsoft.EntityFrameworkCore vs Microsoft.EntityFrameworkCore.InMemory or other EF Core runtime assemblies). The test code itself is correct: it constructs DbContextOptions and calls UseInMemoryDatabase. The real problem is that the test/runtime environment has incompatible NuGet package versions (or missing/old EFCore assemblies) so the required method is not present at runtime.

### Suggested Fix
This is an assembly/package version mismatch issue to fix in the test/project configuration (not a logic bug in the controller). Steps to fix:

1. Ensure all EF Core packages used by both the main project and the test project use the exact same version. In particular verify PackageReference entries for at least:
   - Microsoft.EntityFrameworkCore
   - Microsoft.EntityFrameworkCore.Abstractions (if referenced)
   - Microsoft.EntityFrameworkCore.InMemory
   - Microsoft.EntityFrameworkCore.Relational (if referenced)

   The MissingMethodException indicates the InMemory provider is calling into a method expected in EF Core diagnostics assembly but the runtime assembly does not contain it (version skew). Align versions (e.g., set all to 6.x or 7.x consistently) and restore packages.

2. In the test project (.csproj) add or update the Microsoft.EntityFrameworkCore.InMemory PackageReference to match the version used by the production project. Example (in test project .csproj):

   <PackageReference Include="Microsoft.EntityFrameworkCore.InMemory" Version="<same-version-as-app>" />

3. Clean solution (delete bin/obj), run dotnet restore, rebuild. Optionally run `dotnet list package --outdated` and `dotnet restore` to ensure no mismatched transitive packages.

4. If multiple projects reference EF Core, ensure central package management or floating versions do not introduce mismatches. If necessary, unify versions in Directory.Packages.props or the project files.

5. After package alignment and rebuild the test that calls UseInMemoryDatabase should run without the MissingMethodException. If you cannot change package versions, as a workaround modify tests to avoid UseInMemoryDatabase (e.g., mock DbSets using Moq) but aligning EF Core package versions is the recommended fix.

No production-source code change is required; this is a dependency configuration fix located in project package references.

## EMS.API.UnitTests.csproj - Update_ValidChanges_PersistsAndReturnsOkWithUpdatedEntity
- **Confidence**: High
- **Test File**: ..\EMS.API.UnitTests\Controllers\EmployeeControllerTests.cs
- **Bug Location**: ..\EMS.API.UnitTests\Controllers\EmployeeControllerTests.cs@98

### Analysis
The test fails during construction of the in-memory DbContext with a System.MissingMethodException: Microsoft.EntityFrameworkCore.Diagnostics.AbstractionsStrings.ArgumentIsEmpty(System.Object) not found. The stack trace shows the failure inside EF Core's InMemory UseInMemoryDatabase path (Microsoft.EntityFrameworkCore.InMemory -> Microsoft.EntityFrameworkCore.Utilities.Check.NotEmpty -> AbstractionsStrings). This is a classic runtime binding/version mismatch: different EF Core assemblies (e.g. Microsoft.EntityFrameworkCore.InMemory vs Microsoft.EntityFrameworkCore.* / Diagnostics.Abstractions) are loaded with incompatible versions so the called internal method does not exist. The test code itself (CreateInMemoryContext and the Update test) is logically correct; the exception occurs before any test assertions run. The correct fix is to align EF Core package versions across the solution (the API project and the UnitTests project) so that the InMemory provider and core EF assemblies are the same version and contain the expected methods. Possible causes include: the test project referencing a different EF Core version, or transitive package resolution pulling mixed versions into the test host.

Recommended verification steps:
- Inspect both the API project and UnitTests .csproj PackageReference entries for Microsoft.EntityFrameworkCore, Microsoft.EntityFrameworkCore.InMemory, and related EF packages. Ensure versions match exactly (same major/minor/patch).
- Clean bin/obj folders, run dotnet restore, rebuild solution.
- If using a test runner that loads different assemblies, make sure binding redirects (if applicable) or assembly resolution does not cause older EF assemblies to be loaded.

This is not a logic bug in the test method itself; rather it is an environment/dependency mismatch that prevents the test from running to completion.

### Suggested Fix
This is an assembly/version mismatch issue between EF Core packages used by the test run. Fix by aligning EF Core package versions across projects. For example, in the UnitTests project file ensure PackageReference entries match the main API project: 

1) Edit UnitTests .csproj to use the same EF Core version as the API project. Example (if API uses EF Core 7.0.0):

  <PackageReference Include="Microsoft.EntityFrameworkCore" Version="7.0.0" />
  <PackageReference Include="Microsoft.EntityFrameworkCore.InMemory" Version="7.0.0" />

2) Run:
   dotnet restore
   dotnet clean
   dotnet build

3) Re-run tests. If the solution uses NuGet package consolidation in Visual Studio, use Manage NuGet Packages -> Consolidate to ensure consistent versions.

If after aligning versions the problem persists, inspect the test host's loaded assemblies (fusion logs or test runner diagnostics) for an older EF Core assembly being loaded at runtime and remove/upgrade it. No source change to the test method or controller logic is required.
